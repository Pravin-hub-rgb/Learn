# Doc 4.1.4 — Zod Schema: The Clean Way

## Abhi Tak Kya Hua (4.1.3)

- RHF built-in validation use kiya
- `register()` ke second argument mein rules add kiye
- `required`, `pattern`, `minLength` use kiya
- **Problem:** Bahut messy aur verbose ho gaya!

## Aaj Ka Goal (4.1.4)

**Aaj hum karenge:** Zod use karenge — cleaner, simpler approach

```
Zod install karenge
Schema define karenge — z.string().email(), z.string().min(8)
Type automatically infer karenge — z.infer<typeof schema>
```

---

## Step 1: Problem Recap — RHF Validation Kyun Messy Hai?

**Soch:** "Pichle doc mein humne RHF built-in validation use kiya. Dekho code kitna verbose ho gaya:"

**RHF built-in validation (messy):**
```tsx
register("email", {
  required: "Email is required",
  pattern: {
    value: /^[A-Z0-9._%+-]+@[A-Z0-9.-]+\.[A-Z]{2,}$/i,
    message: "Please enter a valid email address"
  }
})
```

**Problems:**
1. **Regex manually likhna padta hai** — complex, error-prone
2. **Verbose code** — har field ke liye itna code
3. **Type manually define karna padta hai** — `type LoginFormValues = { ... }`
4. **Reusable nahi hai** — har form mein yeh rules repeat karne padenge

**Solution:** Zod use karna — cleaner, simpler, type-safe.

---

## Step 2: Zod Install Karna

**Soch:** "Zod use karne ke liye pehle package install karna padega."

**Zod kya hai:**
- TypeScript-first schema validation library
- Schema define karna easy hai
- Automatic type inference — `z.infer<typeof schema>`
- Built-in validators — `.email()`, `.min()`, `.max()`, etc.

**Install command:**
```bash
npm install zod
```

**Kya install hua:**
- `zod` package — schema validation ke liye
- ~9kb gzipped — lightweight
- No dependencies — koi extra package nahi chahiye

## Step 3: Pehle Login Page Mein Zod Schema Banana

**Soch:** "Zod install ho gaya. Ab schema banana hai. Pehle login page mein hi schema define karta hoon — yahan se start karna easy hoga kyunki main form ke saath hi kaam kar raha hoon. Baad mein isse alag file mein shift kar dunga."

**Kyun pehle login page mein:**
- Login page mein email/password inputs hain
- Wahi validation rules define karne hain
- Pehle yahan test karunga, samjhunga
- Phir alag file mein shift karunga (best practice)

**Zod syntax:**
- `z.object({ field: z.string().rules() })` — Object schema (form fields ke liye)
- `z.string()` — String field (email/password string hain)
- `.email()` — Valid email format (built-in, no regex needed!)
- `.min(8, "message")` — Minimum 8 characters (password ke liye)
- `.min(1, "message")` — Required field (min 1 character)

**Code (login page mein):**
```tsx
"use client";

import { useForm } from "react-hook-form";
import { z } from "zod";

// Login form ke liye Zod schema (pehle yahan define kiya)
const loginSchema = z.object({
  email: z
    .string()
    .min(1, "Email is required")
    .email("Please enter a valid email address"),

  password: z
    .string()
    .min(1, "Password is required")
    .min(8, "Password must be at least 8 characters"),
});

export default function LoginPage() {
  // Abhi sirf schema define kiya hai
  // Baad mein RHF ke saath integrate karenge

  return (
    <div>
      <h1>Login</h1>
      <form>
        {/* Form inputs */}
      </form>
    </div>
  );
}
```

**Explanation:**
- `import { z } from "zod"` — Zod import kiya
- `z.object({ ... })` — Login form ke fields ka schema
- `email: z.string().min(1).email()` — Email validation
- `password: z.string().min(1).min(8)` — Password validation

## Step 4: Schema Ko Alag File Mein Shift Karna

**Soch:** "Schema login page mein define kar diya. Lekin yeh best practice nahi hai — schema ko alag file mein rakhna chahiye taaki reusable ho. `lib/validations/auth.ts` mein shift karta hoon."

**Kyun alag file:**
- Schema reusable hota hai
- Multiple forms mein use kar sakte hain
- Code organized rehta hai
- Type bhi export kar sakte hain

**File structure:**
```
lib/
  validations/
    auth.ts    ← Yahan schemas honge
```

**Code (`lib/validations/auth.ts`):**
```typescript
import { z } from "zod";

// Login form ke liye schema
export const loginSchema = z.object({
  email: z
    .string()
    .min(1, "Email is required")
    .email("Please enter a valid email address"),

  password: z
    .string()
    .min(1, "Password is required")
    .min(8, "Password must be at least 8 characters"),
});

// Type inference
export type LoginInput = z.infer<typeof loginSchema>;
```

**Login page mein import:**
```tsx
import { loginSchema } from "@/lib/validations/auth";
```

## Step 5: Type Inference — Automatic Type!

**Soch:** "Zod ka sabse bada benefit hai automatic type inference. Mujhe manually type define nahi karna padega."

**Type inference kya hai:**
- Zod schema se automatically TypeScript type nikal leta hai
- `z.infer<typeof schema>` — Schema se type infer karta hai
- Manual type define nahi karna padta

**Compare with RHF:**
```tsx
// RHF (manual type - messy):
type LoginFormValues = {
  email: string;
  password: string;
};

// Zod (automatic - clean):
export type LoginInput = z.infer<typeof loginSchema>;
```

**Kya hua:**
- `z.infer<typeof loginSchema>` — Zod schema se automatically type nikal leta hai
- Type aisa hoga: `{ email: string; password: string; }`
- Manual type define nahi karna pada!

**Benefits:**
- Type aur schema ek hi jagah define hote hain
- Schema change kiya toh type automatically update ho jata hai
- DRY principle — Don't Repeat Yourself

## Step 6: Schema Ko Test Karna

**Soch:** "Schema ban gaya, lekin kaam kar raha hai ya nahi verify karna hai. Main ek test file banata hoon jahan main manually test karunga."

**Test file kahan banani hai:**
```
lib/validations/
  auth.ts        ← Yahan schema hai
  auth.test.ts   ← Yahan test code (ya alag test folder mein)
```

**`safeParse` kya hai:**
- `safeParse(data)` — Data validate karta hai, error nahi throw karta
- `result.success` — Boolean, validation pass hua ya nahi
- `result.error.flatten().fieldErrors` — Field-wise error messages

**Test code:**
```typescript
// lib/validations/auth.test.ts (ya koi bhi test file)
import { loginSchema } from "@/lib/validations/auth";

// Test 1: Valid data
const validData = { email: "test@example.com", password: "password123" };
const result1 = loginSchema.safeParse(validData);
console.log("Test 1 (valid):", result1.success);  // true

// Test 2: Invalid email
const invalidEmail = { email: "galat", password: "password123" };
const result2 = loginSchema.safeParse(invalidEmail);
console.log("Test 2 (invalid email):", result2.success);  // false
console.log("Errors:", result2.error.flatten().fieldErrors);

// Test 3: Short password
const shortPassword = { email: "test@example.com", password: "123" };
const result3 = loginSchema.safeParse(shortPassword);
console.log("Test 3 (short password):", result3.success);  // false
```

**Kaise test karna:**
1. Test file create karo (e.g., `lib/validations/auth.test.ts`)
2. Node.js se run karo: `node lib/validations/auth.test.ts`
3. Ya TypeScript se: `npx ts-node lib/validations/auth.test.ts`
4. Console output dekho

**Expected output:**
```
Test 1 (valid): true
Test 2 (invalid email): false
Errors: { email: [ 'Please enter a valid email address' ] }
Test 3 (short password): false
```

**Alternative: Browser console mein test**
- Login page (`/login`) par jao
- Browser console kholo (F12)
- Direct code paste karo:
```javascript
import { loginSchema } from "@/lib/validations/auth";
loginSchema.safeParse({ email: "test@example.com", password: "123" });
```

1. **Zod kyun better hai:**

   - Automatic type inference — `z.infer<typeof schema>`
   - Reusable — ek schema multiple jagah use kar sakte hain

2. **Zod schema syntax:**
   - `z.object({ field: z.string().rules() })`

   - `.min(8, "message")` — Minimum length

3. **Type inference:**
   - `z.infer<typeof schema>` — Automatic type

   - Schema change → type automatically update

4. **Testing with `safeParse`:**
