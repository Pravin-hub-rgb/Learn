# Doc 3.7 — `verifyEmail()` Server Action (Problem-First Approach)

## Context

Ab tak humne:
- User register karta hai → verification email jata hai
- User email mein link par click karta hai → `/verify?token=abc123`
- Verification page (`verify/page.tsx`) load hota hai → `verifyEmail(token)` action call karta hai

Ab humein woh **`verifyEmail()` server action** banana hai jo actually mein token verify kare.

---

## Step 1 — Pehla attempt (jo error dega)

Sabse pehle, hum naturally yeh code likhenge:

File: `app/actions/auth.ts`

```tsx
export async function verifyEmail(token: string) {
  const verificationRecord = await prisma.verificationToken.findUnique({
    where: {
      token,
    },
    include: { User: true },
  });
  // ... aage ka code
}
```

**Logic:** Token se record dhundho, phir verify karo.

Lekin TypeScript immediately error de dega:

```
Type '{ token: string; }' is not assignable to type 'VerificationTokenWhereUniqueInput'.
  Type '{ token: string; }' is not assignable to type '{ id: string; } & ...'.
    Property 'id' is missing in type '{ token: string; }' but required in type '{ id: string; }'.
```

---

## Problem — Yeh error kyun aa raha hai?

`findUnique()` sirf **unique fields** par kaam karta hai. By default, Prisma models mein sirf `@id` field unique hoti hai.

Humare schema mein:

```prisma
model VerificationToken {
  id        String   @id @default(cuid())   // <-- yeh unique hai
  userId    String
  token     String                            // <-- yeh unique NAHI hai
  expiresAt DateTime
  User      User     @relation(fields: [userId], references: [id], onDelete: Cascade)
}
```

`token` field par `@unique` nahi hai, isliye Prisma `findUnique({ where: { token } })` allow nahi karta.

---

## Solution — `token` ko `@unique` banao

Token naturally unique hona chahiye (ek token, ek baar use). Toh schema mein `@unique` add karna sahi bhi hai aur zaroori bhi.

### Step 1 — Schema update karo

File: `prisma/schema.prisma`

```prisma
model VerificationToken {
  id        String   @id @default(cuid())
  userId    String
  token     String   @unique   // <-- Add @unique
  expiresAt DateTime
  User      User     @relation(fields: [userId], references: [id], onDelete: Cascade)
}
```

Terminal mein:

```bash
npx prisma generate
```


### Step 3 — Database push karo

## Step 2 — Ab `verifyEmail()` complete karo

Ab `findUnique({ where: { token } })` kaam karega, kyunki `token` ab unique field hai.

### Complete code:

File: `app/actions/auth.ts`

```tsx
export async function verifyEmail(token: string) {
  // Step 1: Token database mein dhundho
  const verificationRecord = await prisma.verificationToken.findUnique({
    where: {
      token,  // Ab yeh kaam karega kyunki token @unique hai
    },
    include: {
      User: true,
    },
  });

  // Step 2: Token mila hi nahi?
  if (!verificationRecord) {
    return { error: "Invalid verification token" };
  }

  // Step 3: Token expire ho gaya?
  if (verificationRecord.expiresAt < new Date()) {
    // Expired token delete karo (cleanup)
    await prisma.verificationToken.delete({
      where: { id: verificationRecord.id },
    });
    
    return { error: "Verification link expired. Please register again." };
  }

  // Step 4: User ka isVerified = true karo
  await prisma.user.update({
    where: { id: verificationRecord.userId },
    data: { isVerified: true },
  });

  // Step 5: Token delete karo (one-time use only)
  await prisma.verificationToken.delete({
    where: { id: verificationRecord.id },
  });

  // Step 6: Success return karo
  return { success: true };
}
```

## Explanation — Step by step

### Step 1: Token database mein dhundho
```tsx
const verificationRecord = await prisma.verificationToken.findUnique({
  where: { token },
  include: { User: true },
});
```
- `findUnique()` → token unique hai, isliye `findUnique` use kar rahe hain
- `where: { token }` → exact token match dhundho
- `include: { User: true }` → saath mein User ka data bhi le lo (humein `userId` chahiye hoga)

### Step 2: Token valid nahi hai?
```tsx
if (!verificationRecord) {
  return { error: "Invalid verification token" };
}
```
- `!verificationRecord` → agar token database mein nahi mila
- Possible reasons: Token galat hai, already use ho chuka hai, ya kabhi tha hi nahi

### Step 3: Token expire toh nahi hua?
```tsx
if (verificationRecord.expiresAt < new Date()) {
  await prisma.verificationToken.delete({
    where: { id: verificationRecord.id },
  });
  return { error: "Verification link expired. Please register again." };
}
```
- `expiresAt < now` → agar expiry time past mein hai → expired
- Expired token ko delete kar do (cleanup)
- User ko batao ki link expire ho gaya, dobara register kare

### Step 4: User ko verify karo
```tsx
await prisma.user.update({
  where: { id: verificationRecord.userId },
  data: { isVerified: true },
});
```
- User ka `isVerified = true` karo
- Ab yeh user officially "verified" hai

### Step 5: Token ko delete karo (one-time use)
```tsx
await prisma.verificationToken.delete({
  where: { id: verificationRecord.id },
});
```
- Token ek baar hi use ho sakta hai
- Verification ke baad token ko delete kar do
- Agar user dobara same link par click karega → token nahi milega → error milega

### Step 6: Success return karo
```tsx
return { success: true };
```
- Sab kuch successful → verification page success UI dikhayega

## Important Notes

### 1. Error handling order

Humne pehle **token existence** check kiya, phir **expiry** check kiya. Yeh order important hai:
- Pehle batao token hai hi nahi → "Invalid token"
- Phir batao token tha par expire ho gaya → "Link expired"

User ko clear error message milta hai.

### 2. Token deletion timing

Hum token ko **do baar** delete kar rahe hain:
1. **Expiry check ke baad** → agar expired hai toh delete karo
2. **Successful verification ke baad** → one-time use ke liye delete karo

Dono cases mein token delete ho jata hai, jo ki expected behavior hai.

### 3. Return type consistency

Action ka return type:
```tsx
{ success: true } | { error: string }
```

Yeh verification page (`verify/page.tsx`) ke saath match karta hai:
```tsx
if (result.success) {
  setStatus("success");
} else {
  setStatus("error");
}
```

### 4. Security considerations

- Token one-time use hai → delete ho jata hai
- Token 24 ghante mein expire ho jata hai
- Sirf valid token se hi user verify hota hai
- Invalid/expired tokens immediately delete ho jate hain (cleanup)
- Token field bhi unique hai → duplicate tokens impossible

## Testing

Action test karne ke liye:

1. **Valid token test:**
   - Register karo → email verification link milegi
   - Link par click karo → verification page khulega → `verifyEmail()` call hoga
   - Expected: `{ success: true }` → "Email verified successfully!"

2. **Invalid token test:**
   - Manually URL edit karo: `/verify?token=fake123`
   - Expected: `{ error: "Invalid verification token" }`

3. **Expired token test:**
   - 24 ghante baad test karo (ya manually `expiresAt` ko past mein set karo)
   - Expected: `{ error: "Verification link expired. Please register again." }`

4. **Double use test:**
   - Same link do baar click karo
   - Pehli baar: success
   - Doosri baar: `{ error: "Invalid verification token" }` (token delete ho chuka hai)

## Complete Flow Summary

Ab email verification complete ho gaya hai:

1. **Register** → `registerUser()` → User create → Token generate → Email bhejo
2. **Email click** → `/verify?token=abc123` → Verification page load
3. **Verify** → `verifyEmail(token)` → Token validate → User verify → Token delete
4. **Success** → "Email verified! Login karo" → Login page par redirect
