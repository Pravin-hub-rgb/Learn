## Problem: Email Verification Kyun Zaroori Hai?

Soch rahe ho: *"Mera registration system toh ban gaya. User register kar sakta hai. Phir bhi email verification kyun chahiye?"*

**Problem:** Koi bhi fake email se register kar sakta hai!

```
Example:
- User: "abc@xyz.com" (format sahi, lekin email exist nahi karti)
- Password: "123456"
- Result: Database mein fake user create ho gaya! ❌
```

**Consequences:**
- Spam accounts create ho sakte hain
- Fake users database mein pile up ho sakte hain
- Koi bhi kisi aur ke email se register kar sakta hai

**Solution:** Email verification — user ko confirm karna padega ki email actually uski hai.

## Solution: Token-Based Email Verification

### Token Kya Hai?

Token ek **random string** hai (jaise `abc123xyz789`) jo:

1. **Unique hota hai** — har user ka alag token
2. **Ek baar use hota hai** — verify hone ke baad delete ho jata hai
3. **Expiry hoti hai** — 24 hours ke baad kaam nahi karta
4. **Secure hota hai** — token se user ko identify kar sakte hain bina email/password ke

### Token Kahan Store Hota Hai?

Token **do jagah** store hota hai:

**1. Database mein** — `VerificationToken` table mein:
```
VerificationToken table:
id | userId | token | expiresAt
1  | 1      | abc123xyz789 | 2024-01-02 12:00:00
```

**2. URL mein** — Email link mein:
```
https://app.com/verify?token=abc123xyz789
```

  │  ↓                                          │

  └─────────────────────────────────────────────┘

Step 3: Verification Email Bhejna
  ┌─────────────────────────────────────────────┐

Step 4: User Email Check Karta Hai
  ┌─────────────────────────────────────────────┐
  │  User apna email kholta hai                │
  │  ↓                                          │
  │  Email padhta hai: "Click to verify"       │
  │  ↓                                          │
  │  "Verify Email" button/link pe click       │
  └─────────────────────────────────────────────┘

Step 5: Browser /verify Page Pe Jata Hai
  ┌─────────────────────────────────────────────┐
  │  Browser URL open hota hai:                │
  │  http://localhost:3000/verify?token=abc... │
  │  ↓                                          │
  │  /verify page load hoti hai                │
  │  (Next.js Server Component)                │
  │  ↓                                          │

Step 6: Token Verify Karna (Server Action)
  ┌─────────────────────────────────────────────┐

### Token Kyun Use Karte Hain?

**Question:** *"Seedha email verify kyun nahi kar sakte? Token ki kya zaroorat hai?"*

**Answer:** Security aur safety ke liye:

1. **One-time use:** Token ek baar use hone ke baad delete ho jata hai. Agar kisi ne link copy kar liya, toh bhi dobara use nahi kar sakta.

2. **Expiry:** Token ki expiry hoti hai (24 hours). Agar user ne 24 hours mein verify nahi kiya, toh link kaam nahi karega. User ko dobara register karna padega.

3. **No password in URL:** URL mein sirf token hota hai, email/password nahi. Safe hai.

4. **User identification:** Token se hum user ko identify kar sakte hain bina email/password ke.

### Simple Example:

```
User: abc@example.com
Token: abc123xyz789

Email bheji:
  Subject: "Verify your email"
  Content: "Click here to verify: https://app.com/verify?token=abc123xyz789"

User ne click kiya:
  → Browser gaya: /verify?token=abc123xyz789
  → Humne database mein check kiya: token mila? ✓
  → User dhundha: abc@example.com
  → isVerified = true set kiya
  → Token delete kiya

Ab user login kar sakta hai!
```

## API Endpoint Nahi, Server Action Kyun?

**Question:** *"/verify ke liye API endpoint banana padega kya?"*

**Answer:** Nahi! Hum **Server Actions** use kar rahe hain, isliye API endpoint ki zaroorat nahi.

**Old way (API Route):**
```typescript
// app/api/auth/verify/route.ts
export async function POST(request: Request) {
  const { token } = await request.json();
  // ... verify logic
  return Response.json({ success: true });
}
```

**New way (Server Action):**
```typescript
// app/actions/auth.ts
export async function verifyEmail(token: string) {
  // ... verify logic
  return { success: true };
}
```

**Benefits:**
- No REST API needed
- Direct function call from UI
- Type-safe
- Less boilerplate

## Summary

**Aaj humne kya seekha:**

1. **Token kya hai:**
   - Random string (jaise `abc123xyz789`)
   - Unique, one-time use, expiry ke saath
   - Database + URL mein store hota hai

2. **Verification flow:**
   - User register → Token generate → Email bhejo
   - User email verify → Token verify → isVerified = true
   - User login kar sakta hai

3. **Server Actions:**
   - API endpoint ki zaroorat nahi
   - Direct Server Action call
