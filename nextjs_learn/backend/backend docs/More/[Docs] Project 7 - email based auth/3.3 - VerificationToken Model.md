## Problem: Token Ko Store Kahan Karenge?

Soch rahe ho: *"Mujhe verification token generate karna hai aur email mein bhejna hai. Lekin token ko store kahan karun? Kaise verify karunga baad mein?"*

**Requirements:**
1. Token ko database mein save karna hoga
2. Token ke saath user ID bhi store karni hogi (kis user ka token hai)
3. Token ki expiry bhi set karni hogi (24 hours)
4. Baad mein token verify karna hoga

**Solution:** `VerificationToken` table banayenge — jahan tokens store honge.

## Step-by-Step Implementation

### Step 1: Problem Samjho — "Token Kahan Store Hoga?"

Abhi tak humare paas `User` table hai:

```
User table:
id | email | password | isVerified | createdAt
1  | abc@example.com | hashed_pw | false | 2024-01-01
```

Lekin token ko kahan store karenge? User table mein nahi store kar sakte kyunki:
- Ek user ka multiple tokens ho sakte hain (agar multiple times register kiya)
- Token ki expiry hoti hai — alag table mein manage karna behtar hai
- Token verify hone ke baad delete ho jata hai

**Solution:** Alag table — `VerificationToken`

### Step 2: `VerificationToken` Model Banao

*"Sabse pehle, Prisma schema mein naya model add karte hain."*

**Developer ka thought process:**

Soch rahe ho: *"Mujhe ek table chahiye jahan tokens store honge. Prisma model banata hoon."*

```prisma
// prisma/schema.prisma

model VerificationToken {
  id        String   @id @default(cuid())
  userId    String
  token     String   @unique
  expiresAt DateTime
  User      User     @relation(fields: [userId], references: [id], onDelete: Cascade)
}
```

**Model samjho — har field ka reason:**

1. **`id`** — Unique identifier
   - `String` type
   - `@id` — primary key
   - `@default(cuid())` — auto-generate unique ID (CUID format)
   - Har token ka apna unique ID hoga

2. **`userId`** — Kis user ka token hai
   - `String` type
   - Foreign key to `User` table
   - Isse humein pata chalta hai ki yeh token kis user ke liye hai
   - Baad mein user ko dhundhne mein madad karega

3. **`token`** — Verification token (random string)
   - `String` type
   - `@unique` — har token unique hona chahiye
   - Yeh woh random string hai jo email mein bhejenge
   - Example: `abc123xyz789`

4. **`expiresAt`** — Token ki expiry time
   - `DateTime` type
   - 24 hours ki expiry set karenge
   - Agar user ne 24 hours mein verify nahi kiya, toh token kaam nahi karega
   - Security ke liye important

5. **`User` relation** — User table se connection (First time relation dekh rahe ho? Neeche detailed explanation hai!)

   ```prisma
   User User @relation(fields: [userId], references: [id], onDelete: Cascade)
   ```

   **Yeh line confusing lag rahi hai? Chalo break down karte hain:**

   **Pehla `User` — Yeh TYPE hai**
   - Jaise `String`, `DateTime`, `Int` types hote hain
   - Waise hi `User` bhi ek type hai — jo `User` model se aata hai
   - Matlab: yeh field `User` model ka reference/object hold karega

   **Dusra `User` — Yeh FIELD KA NAAM hai**
   - Tum isse kuch bhi rakh sakte ho: `user`, `owner`, `creator`, `author`, etc.
   - Humne `User` rakha kyunki clear hai
   - Example: `user User @relation(...)` bhi likh sakte the
   - Ya `owner User @relation(...)` bhi

   **`@relation(...)` — Yeh DECORATOR hai jo relation define karta hai**
   
   Iske 3 parameters hote hain:

   **a) `fields: [userId]`**
   - Current model (`VerificationToken`) ka kaunsa field foreign key hai
   - `userId` field mein User ka ID store hoga
   - Example: `userId = "cm3qkz5yl0000pe4l3vzq8x9y"`

   **b) `references: [id]`**
   - Dusre model (`User`) ka kaunsa field reference ho raha hai
   - `User.id` se match hoga
   - Matlab: `VerificationToken.userId` → `User.id`

   **c) `onDelete: Cascade`**
   - Agar User delete hua, toh uske tokens bhi auto-delete ho jayenge
   - Options:
     - `Cascade` — delete ho jayenge (humne yeh use kiya)
     - `Restrict` — delete nahi hone denge (error aayega)
     - `SetNull` — `userId` null ho jayega

   **Simple analogy:**
   ```
   Socho tumhare paas ek Aadhar card hai (VerificationToken)
   Aur tumhare papa ka naam likha hai (userId)

   - Pehla User: "Papa" (type) — matlab insaan
   - Dusra User: "Papa" (field name) — tumne isse "Papa" bola
   - @relation: "Yeh mera papa hai, inka Aadhar number XYZ hai"

   Ab jab bhi tum Aadhar card dekhoge:
   - `userId` se sirf papa ka naam pata chalega
   - `User` field se poora papa ka details access kar sakte ho!
   ```

   **Practical example:**
   ```typescript
   // Token fetch kiya
   const token = await prisma.verificationToken.findUnique({...});

   // Ab access kar sakte ho:
   token.userId        // "cm3qkz5yl0000pe4l3vzq8x9y" (sirf ID)
   token.User.email    // "abc@example.com" (poora User object!)
   token.User.isVerified // false

   // Kyunki relation ki wajah se, User object automatically load ho jata hai
   // (agar tum include karo ya select karo)
   ```

   **Relation ka fayda:**
   - Direct User object access kar sakte ho bina alag query ke
   - Code clean rehta hai
   - Type-safe hota hai (TypeScript errors dega agar galat field access karoge)

   - `onDelete: Cascade` — agar user delete hua, toh uske tokens bhi delete ho jayenge

*"Ab Prisma Client regenerate hoga, aur hum TypeScript mein is model ko access kar sakte hain."*

**Example (future use):**

```typescript
// Token save karna
await prisma.verificationToken.create({
  data: {
    userId: "user_123",
    token: "abc123xyz789",
    expiresAt: new Date(Date.now() + 24 * 60 * 60 * 1000), // 24 hours from now
  },
});

// Token dhundhna
const token = await prisma.verificationToken.findUnique({
  where: {
    token: "abc123xyz789",
  },
});

// Expired tokens delete karna
await prisma.verificationToken.deleteMany({
  where: {
    expiresAt: {
      lt: new Date(), // less than current time
    },
  },
});
```
