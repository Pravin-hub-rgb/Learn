### Step 5: onSubmit Handler Add Karo

Ab `onSubmit` function add karo jo `formAction` ko call karega. Yeh function `handleSubmit` ke through call hoga jab form valid hoga:

```tsx
// NEW: onSubmit handler
const onSubmit = async (data: RegistrationFormData) => {
  // FormData banao (same as before)
  const formData = new FormData();
  formData.append("email", data.email);
  formData.append("password", data.password);

  // formAction call karo (useActionState se)
  formAction(formData);
};
```

**Kaise kaam karta hai:**
1. User clicks Submit
2. `handleSubmit` (RHF) validate karta hai (Zod se)
3. Agar valid → `onSubmit` call hota hai
4. `onSubmit` FormData banakar `formAction` ko call karta hai
5. `formAction` → `registerUser` → Server response → `useActionState` state update

### Important: Why Change from `form action` to `form onSubmit`?

**Question:** *"Hum `form action={formAction}` kyun hata rahe hain? Woh toh kaam kar raha tha!"*

**Answer:** Kyunki RHF + Zod use karne ke liye, humein form submission ko **control** karna padta hai.

**`form action={formAction}` (Doc 2.9) — Native HTML Submission:**
1. User clicks Submit
2. Browser automatically FormData collect karta hai
3. Browser directly `formAction` (Server Action) ko call karta hai
4. **Problem:** Client-side validation nahi hoti — form submit hone se pehle kuch check nahi kar sakte

**`form onSubmit={handleSubmit(onSubmit)}` (Doc 2.10) — React-Controlled Submission:**
1. User clicks Submit
2. `handleSubmit` (RHF) pehle validate karta hai (Zod se)
3. Agar valid → `onSubmit` call hota hai
4. `onSubmit` manually FormData banakar `formAction` ko call karta hai
5. **Benefit:** Client-side validation + phir server call

**Comparison:**

| Feature | `form action` | `form onSubmit` |
|---------|---------------|-----------------|
| Validation | Nahi kar sakte | Kar sakte hain (RHF + Zod) |
| Control | Browser controls | React controls |
| Use case | Simple forms | Complex forms with validation |
| Error handling | Server-side only | Client + Server |

**Conclusion:** RHF + Zod ke saath kaam karne ke liye `onSubmit` use karna **zaroori** hai, kyunki humein form submit hone se pehle validate karna hai.

**FormData kya hai:**
- Browser ka built-in object jo form data ko collect karta hai
- Key-value pairs mein data store karta hai
- Server ko bhejne ke liye use hota hai


- Automatically `Content-Type: multipart/form-data` set karta hai
- Server easily `formData.get("email")` se values access kar sakta hai

### Step 6: Complete Flow Samjho

**Kaise kaam kar raha hai:**

```
1. User fills form
   ↓
2. User clicks Submit
   ↓
3. handleSubmit (RHF) pehle validate karta hai (Zod se)
   ↓
   ├── Invalid? → Show field errors → STOP (no server call)
   │
   └── Valid? → onSubmit call hota hai
       ↓
4. onSubmit FormData banata hai
   ↓
5. startTransition(() => formAction(formData)) call hota hai
   ↓
6. useActionState → registerUser(state, formData)
   ↓
7. Server Action database operation karta hai
   ↓
8. Server response → useActionState state update karta hai
   ↓
9. UI re-render → Server error/success message dikhta hai
```

### Important: Why Use `startTransition`?

**Problem:** Jab hum `formAction(formData)` directly call karte hain, toh React warning deta hai:

```
Warning: An async function with useActionState was called outside of a transition.
This is likely not what you intended (for example, isPending will not update correctly).
```

**Kyun?** React ko nahi pata ki yeh update urgent hai ya non-urgent. `useActionState` expects ki action ko **transition** ke andar call kiya jaye.

**React mein do tarah ke updates hote hain:**

1. **Urgent updates** — jo immediately dikhane chahiye
   - Examples: Typing, clicking, hovering
   - UI immediately respond karna chahiye

2. **Non-urgent updates** — jo thoda baad mein bhi dikha sakte hain
   - Examples: Search results, form submission, data fetching
   - Thoda delay acceptable hai

**`startTransition` kya karta hai?**

Yeh React ko batata hai ki:
- "Yeh update non-urgent hai"
- "Pehle urgent updates handle kar lo"
- "Phir isko process karna"

**Analogy:**

Socho React ek chef hai:

**Without `startTransition`:**
```
User clicks Submit
   ↓
Chef immediately drops everything (urgent!)
   ↓
Starts cooking the new dish (state update)
   ↓
Old dish gets thrown away (UI freezes briefly)
```

**With `startTransition`:**
```
User clicks Submit
   ↓
Chef says "Hold on, let me finish this first"
   ↓
Finishes current task (UI stays responsive)
   ↓
Then starts the new dish (non-urgent update)
```

**Code:**

```tsx
import { startTransition } from "react";

// Without startTransition (urgent) - WARNING!
formAction(formData);

// With startTransition (non-urgent) - CORRECT!
startTransition(() => {
  formAction(formData);
});
```

**Benefits:**
- UI responsive rehta hai
- User experience better hota hai
- `isPending` correctly update hota hai
- React warning nahi aati

**Updated `onSubmit` function:**

```tsx
import { startTransition } from "react";

const onSubmit = async (data: RegistrationFormData) => {
  const formData = new FormData();
  formData.append("email", data.email);
  formData.append("password", data.password);

  // React ko batao ki yeh non-urgent update hai
  startTransition(() => {
    formAction(formData);
  });
};
```

### `app/actions/auth.ts` (Same as Doc 2.9)

```typescript
"use server";

import prisma from "@/lib/prisma";
import { hashPassword } from "@/lib/password";

export type RegistrationResult = {
  error?: string;
  success?: boolean;
};

export async function registerUser(
  state: RegistrationResult,
  formData: FormData
): Promise<RegistrationResult> {
  const email = formData.get("email") as string;
  const password = formData.get("password") as string;
  
  // Server-side validation (extra safety)
  if (!email || !password) {
    return { error: "Email and password are required" };
  }
  
  try {
    const hashedPassword = await hashPassword(password);
    
    const newUser = await prisma.user.create({
      data: {
        email,
        password: hashedPassword,
        isVerified: false,
      },
    });
    
    return { success: true };
    
  } catch (error) {
    if (
      typeof error === "object" &&
      error !== null &&
      "code" in error &&
      (error as { code: string }).code === "P2002"
    ) {
      return { error: "Email already exists" };
    }
    
    return { error: "Something went wrong. Please try again." };
  }
}
```
