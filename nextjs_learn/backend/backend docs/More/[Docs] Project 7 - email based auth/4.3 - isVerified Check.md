# Doc 4.3 — isVerified Check

## Abhi Tak Kya Hua (4.2)

- Login form server se connect ho gaya hai
- Zod server side validation kaam kar raha hai
- Database mein user dhoond raha hai
- User nahi mila toh `Invalid credentials` error aa raha hai
- ✅ **Ab ek cheez baaki hai:** User mil bhi gaya toh ab?

## Aaj Ka Goal (4.3)

**Aaj hum karenge:**
✅ `isVerified` field check karenge
✅ Agar user verified nahi hai toh specific error return karenge

> 🎯 is phase mein itna hi hai. Password compare next phase 4.4 mein hoga.

---

## Step 1: Problem Samjho — Ab Kya Baaki Hai?

**Soch:** "Ab user database mein mil bhi gaya hai lekin hume ab ek aur check karna hoga. User ne apna email verify kiya hai ya nahi?"

**Kyun yeh check zaroori hai:**
- Humne Phase 3 mein email verification system banaya tha na?
- Jab user register karta hai tab `isVerified = false` hota hai by default
- Jab woh verification link par click karta hai tab hi `isVerified = true` hota hai
- Unverified users ko login nahi karne dena chahiye

---

## Step 2: Smart Decision — Kisme bhi order sahi karo

🧠 **Bahut important point:**
> Ab tum plan.md mein likha hai password compare karne ke baad check karo.
> 
> ❌ Galat order hai!
> 
> ✅ Sahi order: Pehle `isVerified` check karo, uske baad password compare karo.

**Kyun?**
- `isVerified` sirf ek simple boolean field check hai → 0 cost
- Database se bas 1 byte check karna hai → 0.0001ms lagta hai
- Bcrypt password compare → 100ms+ lagta hai, bahut CPU heavy hota hai

> Agar user hi verified nahi hai toh uske password ko compare karne ka koi fayda nahi hai. Bina zarurat hi nahi hai.

Early exit karo, pehle yeh simple check kar lo.

---

## Step 3: Ab Code Mein Yeh Add Karo

Ab tumhare paas abhi tak yeh code hai:
```typescript
const user = await prisma.user.findUnique({
  where: { email },
});

if (!user) {
  return { error: "Invalid credentials", success: false };
}
```

✅ **Iske neeche bas yeh 3 line add karo:**
```typescript
// ✅ Ab email verified hai ya nahi check karo
if (!user.isVerified) {
  return { error: "Please verify your email first", success: false };
}
```

> 🧠 Ab yeh cheez samjho:
- Yahan pe hum generic error nahi de rahe
- Hum specific error de rahe hai: `"Please verify your email first"
- Kyunki user exist karta hai, password bhi sahi hai lekin bas verify nahi kiya hai
- User ko pata hona chahiye ki usko ab kya karna hai

✅ **Ab poora function ab dikhta hai aise:**
```typescript
"use server";

import prisma from "@/lib/prisma";
import { loginSchema } from "@/lib/validations/auth";

export async function loginUser(prevState, formData) {
  const email = formData.get("email") as string;
  const password = formData.get("password") as string;

  const result = loginSchema.safeParse({ email, password });
  if (!result.success) {
    return { error: result.error.issues[0].message, success: false };
  }

  const user = await prisma.user.findUnique({
    where: { email },
  });

  if (!user) {
    return { error: "Invalid credentials", success: false };
  }

  // ✅ isVerified Check - Phase 3 ka output yahan use ho raha hai ab
  if (!user.isVerified) {
    return { error: "Please verify your email first", success: false };
  }

  console.log("✅ User verified hai:", user.email);

  return { error: "", success: true };
}
```

## Step 4: Ab Test Karo

✅ **2 case test karo:**

### Case 1: **Unverified User**
1.  Naya user register karo
2.  Verification email mein link par click mat karo
3.  Login page par jao
4.  Sahi email + sahi password daalo
5.  Submit karo
6.  ✅ **"Please verify your email first"** error dikhega

### Case 2: **Verified User**
1.  Verification email kholo
2.  Verification link par click karo
3.  Wapas login page par jao
4.  Sahi email + sahi password daalo
5.  Submit karo
6.  ✅ Koi error nahi aayega
7.  Terminal mein `✅ User verified hai:` log dikhega

> ✅ **Yahi hai 4.3 ka end point!**
>
> Ab humne successfully:
> - User verify kar liya hai
> - Verified check kar liya hai
> - Sab valid user ko aage badhane ke liye taiyaar hai
>
> **Agla Step (4.4):** Ab password compare karenge.

---

## Summary

**Aaj humne kya seekha:**

1.  **Order matters:**
    - Pehle sabse cheap operations karo
    - Baad mein expensive operations karo
    - Har baar early exit karo

2.  **Error message strategy:**
    - Jab user exist nahi karta → generic error
    - Jab user exist karta hai lekin verified nahi → specific error

3.  **Phase connection:**
    - Phase 3 mein humne sirf `isVerified` field banaya tha
    - Aaj uska actual use pehli baar hua hai yahan

## Agla Step (4.4)

Phase 4.4 mein hum password compare karenge bcrypt se.
