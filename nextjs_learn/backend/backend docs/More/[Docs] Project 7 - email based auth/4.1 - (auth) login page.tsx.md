## Phase 4 Overview: Login + Session System — Pura Flow

**Bada picture samjho — Phase 4 mein kya hone wala hai:**

### Problem Kya Hai?

Abhi tak (Phase 3 mein) humne email verification system banaya hai. User verify kar sakta hai, `isVerified = true` ho jata hai. Lekin ek problem hai:

**User login kaise kare?**

Current situation:
- User verified hai, lekin login karne ka koi tarika nahi
- Login page nahi hai
- Login form nahi hai
- Session system nahi hai (user ko logged in kaise rakhein?)

**Result:** User verify karne ke baad wapas register page par aa jata hai. Koi dashboard nahi, koi protected content nahi.

### Solution: Login + Session System

Humko yeh banana hai:

**Step-by-Step Flow:**

1. **Login page banao**
   - Simple HTML form
   - Email + password fields
   - "Login" button

2. **Login credentials verify karo**
   - User form submit karta hai
   - `loginUser()` action call hota hai
   - Email se user database mein dhundhta hai
   - Password compare karta hai (`comparePassword()`)
   - `isVerified` check karta hai

3. **Session banao**
   - Credentials valid hain → Session create karo
   - Session database mein save karo
   - Session ID cookie mein set karo
   - User ko redirect karo → `/dashboard`

4. **Dashboard (protected page)**
   - Sirf logged in users dekh sakte hain
   - Session check karo → agar valid hai toh content dikhao
   - Logout button

### Session Kyun Zaroori Hai?

Session ek tarika hai user ko "logged in" rakhne ka. Bina session ke:

```
Problem without session:
- Har page load par user ko dobara login karna padega
- Browser band hua → logout ho gaya
- Koi bhi page kholo → login mangega

Solution with session:
- Ek baar login → browser band karne tak logged in
- Har request par session check → user authenticated
- Protected pages sirf logged in users ke liye
```

### Token vs Session — Fark Samjho

**Phase 3 (Email Verification) mein Token use hua:**
- **Purpose:** Email verify karna
- **Lifetime:** 24 hours (expiry)
- **One-time use:** Ek baar use hone ke baad delete
- **Storage:** Database + URL

**Phase 4 (Login) mein Session use hoga:**
- **Purpose:** User ko logged in rakhna
- **Lifetime:** Days/weeks (long-term)
- **Multi-use:** Har request par use hota hai
- **Storage:** Database + Cookie

## Unverified Users aur Login

**Important:** Login system mein humein `isVerified` check karna hoga.

```typescript
// loginUser() action mein (aage chal kar):

// 1. User dhundo
const user = await prisma.user.findUnique({ where: { email } });

// 2. Pehle isVerified check karo ← PERFORMANCE OPTIMIZATION!
if (!user.isVerified) {
  return { error: "Please verify your email first" };
}

// 3. Phir password check karo (sirf agar verified hai)
const isValid = await comparePassword(password, user.password);

// 4. Session banao
// ... session creation code
```

**Kyun zaroori hai?**
- Agar hum `isVerified` check nahi karenge, toh unverified users bhi login kar payenge
- Phase 3 ka pura email verification system bekaar ho jayega
- Security risk: Koi bhi fake email se register kar ke login kar sakta hai

**Result:** Sirf verified users hi login kar payenge.

### Performance Optimization: `isVerified` Pehle, Password Baad Mein

**Kyun `isVerified` check password se pehle karna chahiye?**

```
Operation Comparison:

isVerified check:
  - Sirf ek boolean comparison hai
  - `if (!user.isVerified)` — instant check
  - CPU cost: negligible ✅

Password compare (bcrypt):
  - Heavy cryptographic computation
  - Multiple rounds of hashing
  - CPU cost: high ❌ (intentionally slow for security)
```

**Order matter karta hai:**

```typescript
// ❌ WRONG ORDER (slow for unverified users)
const user = await prisma.user.findUnique({ where: { email } });
const isValid = await comparePassword(password, user.password); // Heavy!
if (!user.isVerified) {
  return { error: "Please verify your email first" };
}
// Problem: Agar user verified nahi hai, toh humne unnecessary bcrypt compute kiya!

// ✅ CORRECT ORDER (fast exit for unverified users)
const user = await prisma.user.findUnique({ where: { email } });
if (!user.isVerified) {
  return { error: "Please verify your email first" }; // Light check, early exit
}
const isValid = await comparePassword(password, user.password); // Only if verified
// Benefit: Unverified users ke liye expensive bcrypt call hi nahi hogi!
```

**Benefits:**
1. **Performance:** Unverified users ke liye expensive bcrypt computation bach jata hai
2. **Security:** Clear error messages — "verify email first" vs "invalid password"
3. **Resource saving:** Server load kam hota hai (especially under attack)
4. **User experience:** Verified users ko fast response, unverified ko clear guidance

**Real-world impact:**
- Maan lo 100 login attempts aaye
- 50 unverified users ke — bcrypt skip ho gaya (50 × heavy computation bacha!)
- Sirf 50 verified users ke liye hi bcrypt compute hoga

**Yeh ek common optimization pattern hai:**
- Pehle cheap checks (boolean, existence)
- Phir expensive checks (cryptographic operations)
- Early exit se unnecessary work bachta hai

```tsx

"use client";

import { useActionState } from "react";
```
```tsx
import { loginUser } from "@/app/actions/auth";

export default function LoginPage() {

  return (
```

```tsx
            className="border-y border-gray-300 rounded-md shadow-sm focus:outline-none focus:ring-cyan-500 focus:border-cyan-500 w-full px-3 py-1"
            type="email"
            id="email"
            name="email"
            placeholder="Enter your email"
            required
          />
        </div>
        
        <div className="mb-4">
          <label htmlFor="password" className="block text-sm font-medium text-gray-400">
            Password:
          </label>
          <input
            className="border-y border-gray-300 rounded-md shadow-sm focus:outline-none focus:ring-cyan-500 focus:border-cyan-500 w-full px-3 py-1"
            type="password"
            id="password"
            name="password"
            placeholder="Enter your password"
            required
          />
        </div>
        
        <button
          type="submit"
          disabled={isPending}
          className={`w-full py-2 px-4 rounded-md text-white cursor-pointer ${

          Register
        </a>
      </p>
    </div>
  );
}
```
