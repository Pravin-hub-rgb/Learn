# Doc 3.6 — Verification Page (`(auth)/verify/page.tsx`)

## Context

Ab tak humne:
- User register karta hai → verification email jata hai
- Email mein link hoti hai: `https://localhost:3000/verify?token=abc123`

Ab humein woh **verification page** banana hai jo user ko dikhe jab woh us link par click karega.

---

## Problem

User ne email mein verification link par click kiya. Ab kya dikhega?

Humein ek aisa page chahiye jo:
1. URL se `token` parameter nikaale
2. Us token ko verify karne ke liye server action ko bheje
3. Result ke hisaab se user ko feedback de:
   - **Success** → "Email verified! Ab login karo" + login page ka link
   - **Error** → "Invalid ya expired link. Dobara register karo"

---

## Solution

### Step 1 — Page ka structure samjho

Yeh page **client component** hoga kyunki:
- Humein URL se `token` parameter read karna hai (`useSearchParams`)
- Page load hote hi automatically verification trigger karna hai (`useEffect`)
- Verification status (loading/success/error) dikhana hai

### Step 2 — URL se token nikaalo

```tsx
"use client";

import { useSearchParams } from "next/navigation";
import { useEffect, useState } from "react";
import { verifyEmail } from "@/app/actions/auth";
import Link from "next/link";

export default function VerifyPage() {
  const searchParams = useSearchParams();
  const token = searchParams.get("token");
```

**Samjhao:**
- `useSearchParams()` Next.js ka hook hai jo URL query parameters read karta hai
- `searchParams.get("token")` → URL `?token=abc123` mein se `abc123` nikaal ke dega
- `use client` zaroori hai kyunki `useSearchParams` client-side hook hai

### Step 3 — Verification state manage karo

```tsx
  const [status, setStatus] = useState<"loading" | "success" | "error">("loading");
  const [message, setMessage] = useState("");
```

**Samjhao:**
- `status` → page ka current state track karta hai
  - `"loading"` → verification chal raha hai
  - `"success"` → verification successful
  - `"error"` → verification failed
- `message` → user ko dikhane ke liye error/success message

### Step 4 — Page load hote hi verify karo

**Important:** "No token" case ko hum useEffect ke **bahar** handle karenge, kyunki useEffect ke andar directly `setState` call karna cascading renders trigger kar sakta hai (ESLint warning).

Isliye pehle token check karo aur agar token nahi hai toh seedha error state set karo:

```tsx
  // Token nahi hai? Seedha error state set karo (useEffect ke bahar)
  if (!token && status === "loading") {
    setStatus("error");
    setMessage("No verification token provided");
  }

  useEffect(() => {
    // Agar token nahi hai toh useEffect run hi mat karo
    if (!token) return;

    const verifyToken = async () => {
      const result = await verifyEmail(token);
      
      if (result.success) {
        setStatus("success");
        setMessage("Email verified successfully!");
      } else {
        setStatus("error");
        setMessage(result.error || "Verification failed");
      }
    };

    verifyToken();
  }, [token]);
```

**Samjhao:**
- Pehla `if` block → token nahi hai aur abhi tak loading hai? Toh seedha error state set karo
  - `status === "loading"` check zaroori hai taaki har render par setState call na ho (infinite loop se bachao)
- `useEffect` → sirf tab run hoga jab token actually exist karta hai
  - `if (!token) return;` → safety check, useEffect ke andar setState nahi hoga agar token nahi hai
  - Token hai → `verifyEmail(token)` server action call karo
- `verifyEmail` returns:
  - `{ success: true }` → status = "success"
  - `{ error: "..." }` → status = "error"
- `[token]` dependency → agar token change hua toh dobara verify karo (edge case, but safe)

### Step 5 — UI render karo

```tsx
  return (
    <div className="border-x-4 border-gray-600 p-10 w-170 mx-auto mt-10 rounded-2xl">
      <h1 className="text-2xl font-bold mb-4 text-center text-cyan-600">
        Email Verification
      </h1>

      {status === "loading" && (
        <div className="text-center">
          <p className="text-gray-600">Verifying your email...</p>
        </div>
      )}

      {status === "success" && (
        <div className="text-center">
          <p className="text-green-600 mb-4">{message}</p>
          <Link 
            href="/login" 
            className="text-cyan-600 hover:underline"
          >
            Go to Login
          </Link>
        </div>
      )}

      {status === "error" && (
        <div className="text-center">
          <p className="text-red-500 mb-4">{message}</p>
          <Link 
            href="/register" 
            className="text-cyan-600 hover:underline"
          >
            Register Again
          </Link>
        </div>
      )}
    </div>
  );
}
```

**Samjhao:**
- 3 states ke liye 3 alag UI sections
- Har section `status` ke hisaab se render hota hai
- **Loading** → "Verifying your email..."
- **Success** → Green message + login page ka link
- **Error** → Red message + register page ka link

## Complete Code

File: `app/(auth)/verify/page.tsx`

```tsx
"use client";

import { useSearchParams } from "next/navigation";
import { useEffect, useState } from "react";
import { verifyEmail } from "@/app/actions/auth";
import Link from "next/link";

export default function VerifyPage() {
  const searchParams = useSearchParams();
  const token = searchParams.get("token");
  
  const [status, setStatus] = useState<"loading" | "success" | "error">("loading");
  const [message, setMessage] = useState("");

  // Token nahi hai? Seedha error state set karo (useEffect ke bahar)
  // status === "loading" check zaroori hai taaki infinite loop na ho
  if (!token && status === "loading") {
    setStatus("error");
    setMessage("No verification token provided");
  }

  useEffect(() => {
    // Agar token nahi hai toh useEffect run hi mat karo
    if (!token) return;

    const verifyToken = async () => {
      const result = await verifyEmail(token);
      
      if (result.success) {
        setStatus("success");
        setMessage("Email verified successfully!");
      } else {
        setStatus("error");
        setMessage(result.error || "Verification failed");
      }
    };

    verifyToken();
  }, [token]);

  return (
    <div className="border-x-4 border-gray-600 p-10 w-170 mx-auto mt-10 rounded-2xl">
      <h1 className="text-2xl font-bold mb-4 text-center text-cyan-600">
        Email Verification
      </h1>

      {status === "loading" && (
        <div className="text-center">
          <p className="text-gray-600">Verifying your email...</p>
        </div>
      )}

      {status === "success" && (
        <div className="text-center">
          <p className="text-green-600 mb-4">{message}</p>
          <Link 
            href="/login" 
            className="text-cyan-600 hover:underline"
          >
            Go to Login
          </Link>
        </div>
      )}

      {status === "error" && (
        <div className="text-center">
          <p className="text-red-500 mb-4">{message}</p>
          <Link 
            href="/register" 
            className="text-cyan-600 hover:underline"
          >
            Register Again
          </Link>
        </div>
      )}
    </div>
  );
}
```

## Important Notes

### 1. `verifyEmail` action abhi banana baaki hai

Yeh page `verifyEmail(token)` server action ko call karta hai jo abhi humne nahi banaya. Woh hum **Doc 3.7** mein banayenge.

Filhal, agar aap is page ko test karna chahte hain, toh pehle Doc 3.7 complete kar lo.

### 2. `useSearchParams` aur SSR

`useSearchParams` use karne se yeh page **dynamically rendered** ho jata hai (static optimization disable ho jati hai). Iska matlab:
- Har request ke liye page server par render hoga
- Thoda performance impact pad sakta hai

Lekin verification page ke liye yeh acceptable hai kyunki:
- Yeh page kam hi use hoga (sirf verification ke time)
- Humein real-time token read karna zaroori hai

### 3. Token URL mein dikh raha hai

Haan, token URL query parameter mein hai (`?token=abc123`). Yeh:
- Browser history mein save hoga
- Server logs mein dikh sakta hai

Production mein isse behtar approach hoti hai (jaise POST request), lekin simplicity ke liye hum GET use kar rahe hain. Verification tokens one-time use hote hain aur 24 ghante mein expire ho jate hain, isliye security risk minimal hai.

### 4. Link consistency

Register page mein humne verification link aise banayi thi:
```
https://localhost:3000/verify?token=${token}
```

Aur yeh page usi route par hai: `app/(auth)/verify/page.tsx` → `/verify`
