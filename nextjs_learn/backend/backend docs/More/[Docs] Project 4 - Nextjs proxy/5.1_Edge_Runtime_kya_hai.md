# Doc 5.1 — Edge Runtime kya hai

## Abhi Tak Kya Hua

Phase 4 complete — logging, protected routes, rate limiting, geo blocking — sab real scenarios implement kiye. Ab Phase 5 mein samjhenge ki proxy actually **kahan** chalta hai — aur iska kya matlab hai.

---

## Broad Goal — Yeh Doc Mein Kya Karenge

Edge Runtime kya hota hai — normal Node.js server se kaise alag hai — proxy kyun wahan chalta hai.

---

## Pehle samjho — server kahan hota hai?

Normal ek app deploy karo — Vercel ya koi bhi hosting pe — ek server hota hai — ek jagah pe:

```
User India mein hai
        ↓
Request US ke server pe gayi — jahan app deploy hai
        ↓
Response aaya — 200ms lag gaye
```

Dori jitni lambi — utna zyada time.

---

## Edge Network kya hai

Vercel aur Cloudflare jaisi companies ne duniya mein bahut jagah servers lagaye hain — yeh **Edge Nodes** hain:

```
Mumbai, Singapore, Frankfurt, New York, São Paulo...
```

Jab Edge pe deploy karo — user ke sabse **kareeb wale** node pe chalta hai code:

```
User India mein hai
        ↓
Mumbai Edge Node pe request gayi — kareebi server
        ↓
Response aaya — 20ms lag gaye  ← bahut fast
```

Yahi hai **Edge Runtime** — code edge nodes pe chalta hai — user ke paas — fast.

---

## Proxy Edge Runtime mein chalta hai — kyun?

Proxy ki kaam kya hai? — Har request pe chalo — check karo — redirect karo — fast.

Agar proxy US ke server pe hota — aur user India mein — pehle US jaata request — proxy check karta — phir wapas India — 400ms+.

Edge pe — Mumbai node pe — 5ms mein check ho jaata.

Isliye proxy by default Edge Runtime mein chalta hai — fast redirects ke liye.

---

## Edge Runtime ka fark normal Node.js se

Node.js mein jo bhi chahe use kar sakte hain — Edge Runtime mein restrictions hain:

```
Node.js Runtime         Edge Runtime
─────────────────────────────────────────
File system access      ❌ Nahi
Native Node modules     ❌ Nahi
(fs, path, crypto...)
Database connections    ❌ Directly nahi
Heavy libraries         ❌ Size limit hai
fetch()                 ✅ Available
Web APIs                ✅ Available
Cookies, Headers        ✅ Available
```

**Kyun restrictions?** — Edge nodes lightweight hone chahiye — fast startup — har jagah deploy ho sake — full Node.js environment nahi hota wahan.

---

## Practically kya matlab hai?

Proxy mein yeh **nahi** kar sakte:

```ts
// ❌ File system
import fs from "fs"
fs.readFileSync("./data.json")

// ❌ Heavy database client directly
import { PrismaClient } from "@prisma/client"
const prisma = new PrismaClient()
const user = await prisma.user.findUnique(...)

// ❌ bcrypt — native module use karta hai
import bcrypt from "bcrypt"
```

Proxy mein yeh **kar sakte** hain:

```ts
// ✅ fetch — external API call
const res = await fetch("https://api.example.com/verify")

// ✅ Cookies, headers padho
request.cookies.get("session")

// ✅ Redirect, rewrite
NextResponse.redirect(...)

// ✅ JWT verify — pure JS library
import { jwtVerify } from "jose"
```

---

## Summary

- Edge Runtime — user ke paas wale servers pe code chalta hai — fast
- Proxy by default Edge Runtime mein — isliye quick redirects
- File system, native modules, heavy databases — Edge mein nahi
- fetch, cookies, headers, redirect — sab available
