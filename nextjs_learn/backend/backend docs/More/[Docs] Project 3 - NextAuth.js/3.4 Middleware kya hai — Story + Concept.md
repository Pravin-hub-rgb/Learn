# Doc 3.4 — Middleware kya hai — Story + Concept

## Abhi Tak Kya Hua
3.3 mein `/dashboard` ko Server Component protection di — `auth()` + `redirect()` — sirf 2 lines — koi flash nahi — server pe hi check hota hai.

---

## Broad Goal — Yeh Doc Mein Kya Karenge

3.3 wali approach mein ek problem hai — aaj discover karenge. Phir uska solution — **Middleware** — story se samjhenge. Koi code nahi aaj — sirf concept.

---

## 3.3 ki limitation — ek problem hai

Abhi `/dashboard` protected hai — `auth()` + `redirect()` se.

Par socho — kal ek naya page banana hai `/profile` — woh bhi sirf logged-in users ke liye. Aur `/settings` — woh bhi. Aur `/orders` — woh bhi.

Har page mein yeh likhna padega:

```tsx
// /profile/page.tsx
const session = await auth()
if (!session) redirect("/")

// /settings/page.tsx
const session = await auth()
if (!session) redirect("/")

// /orders/page.tsx
const session = await auth()
if (!session) redirect("/")
```

**Har page mein same 2 lines** — copy paste karte rehna. 10 pages hain toh 10 jagah likhna. Kisi ek jagah bhool gaye — woh page unprotected reh jaayega.

Yeh approach scale nahi karti.

---

## Behtar kya hoga?

Socho ek building hai — andar jaane ke liye har room mein alag alag guard lagaya hua hai. Koi bhi pehle seedha room mein jaata hai — wahan guard check karta hai — agar permission nahi toh bahar karta hai.

Yeh wahi hai jo abhi kar rahe hain — har page pe check.

**Better solution** — building ke **main gate pe ek guard lagao** — andar jaane se pehle hi check ho jaaye — koi bhi unauthorized room tak pahunche hi nahi.

Yahi hai **Middleware.**

---

## Middleware kya hai

Middleware ek function hai jo **har request pe — page render hone se pehle — chalta hai.**

```
User koi bhi URL visit karta hai
        ↓
Middleware pehle chalta hai — check karta hai
        ↓
"Allow karo" ya "Redirect karo" decide karta hai
        ↓
Phir page render hota hai — ya redirect hota hai
```

Har page mein kuch likhna nahi — ek jagah sab handle.

---

## Real life analogy — airport security

Airport pe socho:

```
Tu flight pakadne jaata hai
        ↓
Security check pehle — boarding pass, ID
        ↓
Pass hua — andar jao — flight gate tak
        ↓
Fail hua — bahar — flight nahi milegi
```

Har flight ke gate pe alag security nahi hoti — ek hi security check — sab handle karta hai.

**Middleware = Airport security**
**Pages = Flight gates**
**User = Passenger**

## Next.js mein Middleware kahan hota hai?

Next.js mein middleware ke liye ek special file hoti hai — root mein — `package.json` ke saath.

Pehle log `middleware.ts` likhte the — aur documentation mein abhi bhi yahi milega:

```
nextauth-project/
├── app/
├── middleware.ts   ← purana tarika
├── auth.ts
├── package.json
```

Par agar tu Next.js 16 mein `middleware.ts` banayega — terminal mein yeh warning aayegi:

```
⚠ The "middleware" file convention is deprecated. 
Please use "proxy" instead.
```

Next.js 16 ne rename kar diya — ab `proxy.ts` likhna hai:

```
nextauth-project/
├── app/
├── proxy.ts        ← naya naam — same kaam
├── auth.ts
├── package.json
```

Same code — sirf file ka naam alag. Isliye agar kahin documentation mein `middleware.ts` dikhe — toh samajh lo `proxy.ts` likhna hai.

---

## Middleware kab chalta hai — exactly?

Har request pe — **page render hone se pehle:**

```
localhost:3000/dashboard visit kiya
        ↓
proxy.ts chala pehle
        ↓
"Logged in hai?" — check kiya
        ↓
Nahi → redirect to "/"  ← page render hi nahi hua
Haan → dashboard render hua
```

Yeh Server Component se bhi ek step pehle hai — Server Component render hone se pehle Middleware chalta hai.

---

## Middleware ka flow — teeno approaches compare karo

```
Client-side (3.1):
Browser → Page load → JS chala → Check → Redirect
❌ Flash — content dikh jaata hai

Server Component (3.3):
Browser → Server → Page render shuru → auth() → Redirect
✅ No flash — par har page mein likhna padta hai

Middleware (3.5):
Browser → proxy.ts → Check → Redirect (ya allow)
✅ No flash — ek jagah sab handle — har page pe kuch nahi likhna
```
