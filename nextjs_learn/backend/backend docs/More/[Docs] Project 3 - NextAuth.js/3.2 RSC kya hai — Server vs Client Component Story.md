# Doc 3.2 — RSC kya hai — Server vs Client Component Story

## Abhi Tak Kya Hua
3.1 mein `/dashboard` page banaya — `useSession()` + `useEffect` + `useRouter` se client-side protection lagayi. Kaam karta hai — par ek badi limitation thi:

> "Server ko pata hi nahi tha ki user logged in nahi tha — usne page bhej diya — phir browser ne redirect kiya."

Aaj yeh samjhenge ki server ko pehle se kaise pata chal sakta hai — taaki page bhejne se pehle hi check ho jaaye.

---

## Broad Goal — Yeh Doc Mein Kya Karenge

React Server Components (RSC) ka concept samjhenge — Server Component aur Client Component mein kya fark hai — kab konsa use karte hain — aur kyun server pe check karna better hai.

Koi naya code nahi aaj — sirf concept clear karenge — taaki 3.3 mein server-side protection naturally samajh aaye.

---

## Pehle socho — website kaise kaam karti hai normally?

Purani websites mein — jab tu koi page kholta tha:

```
Tu browser mein URL type karta hai
        ↓
Server pe request jaati hai
        ↓
Server poora HTML banata hai
        ↓
Browser ko bhejta hai
        ↓
Browser sirf HTML dikhata hai
```

Sab kuch server pe hota tha — browser sirf display karta tha. Isme ek problem thi — **interactivity nahi thi.** Button click karo — kuch nahi hoga — jab tak dobara server se request nahi aayi.

---

## Phir React aaya — sab client pe

React ne sab kuch browser mein shift kar diya:

```
Browser pe JavaScript load hota hai
        ↓
React saara UI browser mein banata hai
        ↓
Button click — React state change — UI update
        ↓
Sab kuch browser mein — server se baat nahi
```

Interactivity aa gayi — par ek naya problem:

**Pehli baar page load hone mein time lagta hai** — JavaScript pehle load ho — phir React chale — phir UI bane. Tab tak user blank screen dekh raha hota hai.

Aur ek aur problem — **sensitive kaam bhi browser mein hone laga** — database calls, auth checks — sab browser mein. Yeh secure nahi.

---

## Next.js ne dono worlds mein balance banaya — RSC

Next.js App Router mein **React Server Components** aaye — idea simple hai:

> "Jo cheez server pe ho sakti hai — server pe karo. Jo cheez browser mein honi chahiye — woh browser mein karo."

**Server Component** — server pe run hota hai:
```
Request aayi
        ↓
Server pe component ka code chala
        ↓
HTML ban gaya
        ↓
Browser ko HTML bheja — React code nahi
        ↓
Browser dikhata hai — instantly
```

**Client Component** — browser mein run hota hai:
```
Server se HTML aaya
        ↓
Browser mein JavaScript load hua
        ↓
Component browser mein "hydrate" hua
        ↓
Ab interactive hai — onClick, useState sab kaam karta hai
```

---

## Kab Server Component — kab Client Component?

Ek simple rule:

**Server Component use karo jab:**
- Data fetch karna ho — database, API
- Auth check karni ho — session padhni ho
- Sensitive kaam — jo browser mein nahi hona chahiye
- Heavy logic — jo browser pe slow hoga

**Client Component use karo jab:**
- `useState` chahiye — koi cheez "yaad" rakhni ho
- `useEffect` chahiye — render ke baad kuch karna ho
- Browser events — `onClick`, `onChange`, `onSubmit`
- Browser-only APIs — `localStorage`, `window`

---

## Default kya hai?

**Next.js App Router mein har component by default Server Component hai.**

Jab tak file ke upar `"use client"` na likha ho — woh Server Component hai.

```tsx
// Server Component — default
export default function Page() {
  return <h1>Hello</h1>
}

// Client Component — explicitly batana padta hai
"use client"
export default function Navbar() {
  const [open, setOpen] = useState(false)
  return <nav>...</nav>
}
```

---

## Yaad karo — humne yeh pehle bhi kiya hai

`page.tsx` mein `auth()` use kiya tha — remember?

```tsx
// page.tsx — "use client" nahi likha — Server Component hai
export default async function Home() {
  const session = await auth() // ← server pe cookie padhi
  // ...
}
```

Yeh Server Component tha — isliye `auth()` kaam kiya — server pe cookie directly padh sakte hain.

Aur `Navbar.tsx` mein `"use client"` likha tha — isliye `useSession()` use kiya — kyunki browser mein session API call se aata hai.

**Dots connect ho gaye?** ✅

---

## Server Component mein auth check — kyun better hai?

3.1 mein client-side protection ka flow tha:

```
Browser pe /dashboard load hua
        ↓
JavaScript chala
        ↓
useSession() ne server se session fetch kiya
        ↓
"unauthenticated" — redirect
```

**Problem:** Server ne page bhej diya — browser ne baad mein check kiya.

Server Component mein flow hoga:

```
/dashboard pe request aayi
        ↓
Server pe auth() chali — session check hua
        ↓
"unauthenticated" — server ne redirect kar diya
        ↓
Browser tak page pahuncha hi nahi
```

**Koi flash nahi — koi JavaScript nahi — server ne hi rok diya.**

Yeh 3.3 mein implement karenge.

---

## Summary

- Purani websites — sab server pe — interactivity nahi
- React — sab browser mein — slow first load, security issues
- RSC — balance — server pe jo ho sake, browser mein jo zaroori ho
- Default — Server Component — `"use client"` se Client Component
- Server Component — `auth()` directly — fast, secure
- Client Component — `useSession()` — browser mein, API call se
- Server pe check — page bhejne se pehle — no flash, no JavaScript needed

## Agla Step

**Doc 3.3** mein `/dashboard` page ko Server Component protection se secure karenge — `auth()` se session check karenge — server pe hi redirect karenge — 3.1 wali saari limitations khatam ho jaayengi.
