# Doc 4.2 — Error Handling Login Page pe

## Abhi Tak Kya Hua

4.1 mein custom `/login` page banaya — `auth.ts` mein `pages.signIn: "/login"` add kiya — Navbar ka `signIn()` fix kiya — middleware bhi `/login` pe redirect karne laga. Abhi login page pe sirf ek button hai — "Sign In with GitHub".

---

## Broad Goal — Yeh Doc Mein Kya Karenge

Ek cheez missing hai abhi — agar login mein **kuch galat ho jaaye** toh user ko pata hi nahi chalta. Page chup rehta hai — koi message nahi.

Aaj yeh fix karenge — do kaam honge:
- Pehle samjhenge ki Next-Auth error kahan dalta hai — URL mein
- Phir woh URL se error padhenge — aur page pe dikhayenge

---

## Next-Auth error kahan dalta hai?

Jab Next-Auth ko koi problem aati hai login mein — woh ek cheez karta hai — **URL mein error ka naam daal deta hai:**

```
/login?error=OAuthAccountNotLinked
```

Yeh `?error=OAuthAccountNotLinked` wala part — **query parameter** kehte hain isse.

Query parameter kya hota hai? — URL ke end mein `?` ke baad jo kuch hota hai. Jaise Google pe search karo:

```
google.com/search?q=nextjs
                  ↑
             query parameter — q = "nextjs"
```

Same tarah Next-Auth ne likha:

```
/login?error=OAuthAccountNotLinked
        ↑
   query parameter — error = "OAuthAccountNotLinked"
```

**Yeh error kab aayegi actually?**

Abhi hamare app mein sirf GitHub provider hai — toh real mein trigger karna mushkil hai. Par jab baad mein Google bhi add karoge — tab yeh scenario hoga:

```
User ne pehle GitHub se login kiya — email: abc@gmail.com
        ↓
Phir Google se login karne ki koshish ki — same email: abc@gmail.com
        ↓
Next-Auth confuse — "yeh email toh GitHub se linked hai — Google se kaise?"
        ↓
Error diya — /login?error=OAuthAccountNotLinked pe redirect kiya
```

Abhi ke liye hum **manually URL mein error daalke** test karenge — concept same hai — jab real error aayegi tab bhi yahi URL format hoga.

Browser mein yeh URL visit karo:

```
http://localhost:3000/login?error=OAuthAccountNotLinked
```

Page bilkul same dikhega — koi fark nahi — error message nahi.

**Kyun?** — URL mein error hai — par login page us URL ko padh hi nahi raha. Yeh fix karna hai.

---

## `searchParams` — URL padhne ka tarika

Next.js mein har page component ko ek cheez automatically milti hai — **`searchParams`**.

Pehle simple example se samjho — login se bilkul alag:

Socho ek page hai — aur URL hai:
```
/products?category=shoes&color=red
```

Next.js automatically yeh object banata hai:
```js
searchParams = {
  category: "shoes",
  color: "red"
}
```

Aur yeh object **har page component ko automatically milta hai** — bas function ka parameter banana padta hai.

Kaise milta hai? — Normal function parameters ki tarah:

```tsx
// Pehle bina searchParams ke — yeh abhi hai hamare paas
export default function LoginPage() {
  return <main>...</main>
}

// Ab searchParams ke saath — Next.js automatically dega
export default function LoginPage({ searchParams }) {
  console.log(searchParams)
  return <main>...</main>
}
```

Yeh `{ searchParams }` — **destructuring** hai. Next.js ek bada object deta hai page component ko — usme se sirf `searchParams` wala part nikal rahe hain.

Simple example destructuring ka:

```js
// Bina destructuring — poora object milta hai
function example(props) {
  console.log(props.searchParams)
}

// Destructuring ke saath — seedha nikal lo
function example({ searchParams }) {
  console.log(searchParams) // same cheez — seedha
}
```

## TypeScript wala part — `{ error?: string }`

Abhi `searchParams` ke saath TypeScript ka type bhi likhna hoga — warna TypeScript error dega ki "yeh kya hai, iska type batao."

Type matlab — "is cheez mein kya kya hoga" — TypeScript ko pehle se batao.

```tsx
export default function LoginPage({
  searchParams,
}: {
  searchParams: { error?: string }
}) {
```

Yeh `}: { searchParams: { error?: string } }` wala part — sirf TypeScript ko bata raha hai:

```
searchParams ek object hai
  jisme ek field hai — error
  woh field string hogi
  ? matlab — optional hai — ho bhi sakta hai, nahi bhi
```

`?` kyun — kyunki normal `/login` pe koi error nahi hogi — `?error=...` nahi hoga URL mein — toh `error` field hogi hi nahi `searchParams` mein. Optional isliye.

## Step 1 — `searchParams` se error padho

Ab `login/page.tsx` mein `searchParams` add karo — pehle sirf `console.log` karo — dekho kya milta hai:

```tsx
import { signIn } from "@/auth"

export default function LoginPage({
  searchParams,
}: {
  searchParams: { error?: string }
}) {
  console.log(searchParams)

  return (
    <main>
      <h1>Sign In</h1>
      <form
        action={async () => {
          "use server"
          await signIn("github")
        }}
      >
        <button type="submit">Sign In with GitHub</button>
      </form>
    </main>
  )
}
```

**Note:** Next.js 15+ mein `searchParams` ek Promise ban gaya hai. Isliye seedha use karne par error aata hai:
```
Route "/login" used `Object.keys(searchParams)` or similar. `searchParams` is a Promise and must be unwrapped with `await` or `React.use()` before accessing its properties.
```

**Sahi tarika:**
```tsx
// ✅ Sahi tarika - function ko async banana padega
export default async function LoginPage({
  searchParams,
}: {
  searchParams: { error?: string }
}) {
  console.log(await searchParams)  // ← Ab await use karenge
  // ...
}
```

**Terminal** mein dekho — yeh URL visit karo:

```
http://localhost:3000/login?error=OAuthAccountNotLinked
```

Terminal mein print hoga:
```js
{ error: 'OAuthAccountNotLinked' }
```

Ab normal `/login` visit karo:
```js
{}
```

Error nahi hai toh empty object. ✅ — `searchParams` kaam kar raha hai.

## Step 2 — error variable mein nikalo

`searchParams.error` se `error` nikal lo — `console.log` hata do:

```tsx
import { signIn } from "@/auth"

export default async function LoginPage({
  searchParams,
}: {
  searchParams: { error?: string }
}) {
  const error = (await searchParams).error

  return (
    <main>
      <h1>Sign In</h1>
      <form
        action={async () => {
          "use server"
          await signIn("github")
        }}
      >
        <button type="submit">Sign In with GitHub</button>
      </form>
    </main>
  )
}
```

**Explanation:** `searchParams` ek Promise hai jo URL ke query parameters ko represent karta hai. Jab hum `await searchParams.error` likhte hain, toh hum Promise ke andar ke property ko directly access karne ki koshish karte hain, jo ki galat hai.

Sahi tarika hai pehle Promise ko resolve karna (`await searchParams`), phir uske baad property access karna (`.error`).

```tsx
// ❌ Galat tarika - searchParams Promise hai
const error = await searchParams.error  // ← Yeh galat hai

// ✅ Sahi tarika - pehle Promise resolve karo
const error = (await searchParams).error  // ← Yeh sahi hai
```

Visual example:
```tsx
// Promise resolve hone ke baad object milta hai
const resolvedParams = await searchParams  // → { error: "OAuthAccountNotLinked" }
const error = resolvedParams.error           // → "OAuthAccountNotLinked"
```

`const error = searchParams.error` — `searchParams` object ke andar se `error` field nikal li.

```
URL: /login?error=OAuthAccountNotLinked  →  error = "OAuthAccountNotLinked"
URL: /login                              →  error = undefined
```

`undefined` — matlab woh cheez exist hi nahi karti. Jab URL mein `?error=...` nahi — toh `error` field hi nahi hogi — isliye `undefined`.

## Step 3 — error page pe dikhao

`error` variable hai — ab page pe dikhao. `{error && ...}` — yeh pattern yaad hai 2.5 se? — "agar `error` hai tabhi dikhao":

```tsx
import { signIn } from "@/auth"

export default async function LoginPage({
  searchParams,
}: {
  searchParams: { error?: string }
}) {
  const error = (await searchParams).error

  return (
    <main>
      <h1>Sign In</h1>

      {error && (
        <p style={{ color: "red" }}>
          Kuch galat hua — dobara try karo
        </p>
      )}

      <form
        action={async () => {
          "use server"
          await signIn("github")
        }}
      >
        <button type="submit">Sign In with GitHub</button>
      </form>
    </main>
  )
}
```

Browser mein dekho:

```
http://localhost:3000/login?error=OAuthAccountNotLinked
```
→ Red message dikh raha hai? ✅

```
http://localhost:3000/login
```
→ Koi message nahi? ✅

## Step 4 — error ke hisaab se specific message dikhao

Abhi sirf ek generic message hai — "Kuch galat hua". Par Next-Auth kai alag errors deta hai — user ko specific batana chahiye.

Common errors:

```
OAuthAccountNotLinked  → same email pehle se kisi aur provider se linked hai
OAuthSignin            → GitHub se connect karne mein problem aayi
OAuthCallback          → GitHub ne callback mein kuch galat bheja
Default                → koi aur unknown error
```

Ek object banao — har error ka message. Object component ke bahar banao — kyunki yeh sirf data hai — component ke andar kuch nahi hota isse:

```tsx
const errorMessages = {
  OAuthAccountNotLinked: "Yeh email pehle se kisi aur account se linked hai",
  OAuthSignin: "GitHub se connect karne mein dikkat aayi — dobara try karo",
  OAuthCallback: "Login process mein kuch gadbad hui — dobara try karo",
  Default: "Kuch galat hua — dobara try karo",
}
```

Yeh ek normal object hai — key: value. Key hai error ka naam — value hai uska human-readable message.

Ab is object se sahi message nikalo. Pehle simple example se samjho:

```js
const messages = {
  hello: "Namaste",
  bye: "Alvida",
}

const word = "hello"
console.log(messages[word]) // → "Namaste"
```

`messages[word]` — `word` ki value se object mein se message nikala. Same cheez karenge `error` se.

Par ek problem — agar `error` object mein hai hi nahi toh kya hoga?

```js
console.log(messages["kuchBhi"]) // → undefined
```

`undefined` aayega — koi message nahi dikhega. Isliye **fallback** chahiye — "agar yeh error object mein nahi mili — toh Default wala dikhao."

Yahan `??` aata hai — **nullish coalescing operator**. Pehle simple example:

```js
const result = undefined ?? "fallback"
console.log(result) // → "fallback"

const result2 = "actual value" ?? "fallback"
console.log(result2) // → "actual value"
```

`??` matlab — "agar left side `null` ya `undefined` hai — toh right side use karo — warna left side use karo."

Ab apply karo:

```tsx
const errorMessage = error
  ? errorMessages[error] ?? errorMessages["Default"]
  : null
```

Teen parts hain:

```
error                           → kya URL mein error hai?
  ? errorMessages[error]        → haan — toh us error ka message nikalo object se
    ?? errorMessages["Default"] → agar woh error object mein nahi mila — Default lo
  : null                        → URL mein error hi nahi — toh null
```

Ab poora `login/page.tsx`:

```tsx
import { signIn } from "@/auth"

const errorMessages = {
  OAuthAccountNotLinked: "Yeh email pehle se kisi aur account se linked hai",
  OAuthSignin: "GitHub se connect karne mein dikkat aayi — dobara try karo",
  OAuthCallback: "Login process mein kuch gadbad hui — dobara try karo",
  Default: "Kuch galat hua — dobara try karo",
}

export default async function LoginPage({
  searchParams,
}: {
  searchParams: { error?: string }
}) {
  const error = (await searchParams).error
  const errorMessage = error
    ? errorMessages[error] ?? errorMessages["Default"]
    : null

  return (
    <main>
      <h1>Sign In</h1>

      {errorMessage && (
        <p style={{ color: "red" }}>
          {errorMessage}
        </p>
      )}

      <form
        action={async () => {
          "use server"
          await signIn("github")
        }}
      >
        <button type="submit">Sign In with GitHub</button>
      </form>
    </main>
  )
}
```

**Test karo:**

```
http://localhost:3000/login?error=OAuthAccountNotLinked
```
→ "Yeh email pehle se kisi aur account se linked hai" ✅

```
http://localhost:3000/login?error=KoiBhiRandomError
```
→ "Kuch galat hua — dobara try karo" ✅

```
http://localhost:3000/login
```
→ Koi message nahi ✅

## TypeScript Error — "Element implicitly has an 'any' type"

Jab aap `errorMessages[error]` likhte hain, toh TypeScript ek error dega:

```
Element implicitly has an 'any' type because expression of type 'string' can't be used to index type '{ OAuthAccountNotLinked: string; OAuthSignin: string; OAuthCallback: string; Default: string; }'.
  No index signature with a parameter of type 'string' was found on type '{ OAuthAccountNotLinked: string; OAuthSignin: string; OAuthCallback: string; Default: string; }'.
```

**Problem:** TypeScript ko nahi pata ki `error` string hai jo `errorMessages` object ke keys ke match karti hai. TypeScript sochta hai ki `error` koi bhi string ho sakti hai, aur `errorMessages` mein sirf specific keys hain.

**Solution:** Type assertion use karo:

```tsx
// ❌ TypeScript error aayega
const errorMessage = error ? errorMessages[error] : null;

// ✅ Type assertion use karo
const errorMessage = error
  ? (errorMessages[error as keyof typeof errorMessages] ?? errorMessages["Default"])
  : null;
```

**Explanation:** `error as keyof typeof errorMessages` TypeScript ko batata hai ki `error` string hai jo `errorMessages` object ke keys ke match karti hai. `keyof typeof errorMessages` matlab "errorMessages object ke sabhi keys".

**Visual example:**
```tsx
// errorMessages ke keys hain: "OAuthAccountNotLinked", "OAuthSignin", "OAuthCallback", "Default"
// error variable mein URL se aayi string hai: "OAuthAccountNotLinked"

// TypeScript ko batana padega ki error string hai jo errorMessages ke keys ke match karti hai
const errorMessage = errorMessages[error as keyof typeof errorMessages];
```

Ab TypeScript error nahi dega aur code sahi se kaam karega.

## Summary

- Next-Auth errors URL mein query parameter ke roop mein dalta hai — `/login?error=...`
- `searchParams` — Next.js har page component ko automatically deta hai — URL ke query params
- Destructuring — `{ searchParams }` se function parameter seedha nikal liya
- `??` — nullish coalescing — agar left side `undefined` ya `null` toh right side use karo
- Error ke hisaab se specific message dikhaya
