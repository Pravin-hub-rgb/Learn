## Pehle Socho — Ek Todo Mein Kya Kya Hoga?

Seedha code mein mat jaao — practically socho.

"Todo app banana hai. Toh ek todo mein kya information hogi?"

**Title hoga** — "Buy milk", "Call mom" — koi text.

**Done hoga** — complete hua ya nahi — `true` ya `false`.

**Id hogi** — do todos same title ke saath ho sakte hain. Inhe alag kaise pehchano? Id se — har todo ka unique number — automatically 1, 2, 3...

Bas. **Teen cheezein — `id`, `title`, `done`.**

```
id     → 1
title  → "Buy milk"
done   → false
```

## `schema.prisma` Mein Todo Model Likho

`prisma/schema.prisma` kholo. Neeche ek khali model banao:

```prisma
model Todo {

}
```

Iske andar teen cheezein add karni hain — ek ek karke.

---

### `id` Add Karo

Id ek integer hai — `Int` type:

```prisma
model Todo {
  id Int
}
```

Do cheezein aur batani hain —

Yeh primary key hai — unique identifier. `@id` lagao:

```prisma
model Todo {
  id Int @id
}
```

Automatically 1, 2, 3... badhta jaaye — manually nahi dena. `@default(autoincrement())` lagao:

```prisma
model Todo {
  id Int @id @default(autoincrement())
}
```

`@default()` matlab — "jab naya todo bane aur yeh field na di jaaye, toh yeh value use karo."

`autoincrement()` matlab — "khud se 1 se shuru karke badhte jao."

---

### `title` Add Karo

Text hai — `String` type:

```prisma
model Todo {
  id    Int    @id @default(autoincrement())
  title String
}
```

Koi `@default` nahi — title user dega. Nahi diya toh Prisma error dega — sahi hai.

---

### `done` Add Karo

`true` ya `false` — `Boolean` type:

```prisma
model Todo {
  id    Int     @id @default(autoincrement())
  title String
  done  Boolean
}
```

Naya todo hamesha incomplete hoga — `@default(false)`:

```prisma
model Todo {
  id    Int     @id @default(autoincrement())
  title String
  done  Boolean @default(false)
}
```

Model ready.

## Abhi Schema Likha — Par Database Mein Table Nahi Bani

`schema.prisma` mein model likhne se database mein kuch nahi hota — yeh sirf blueprint hai kaagaz pe.

Neon mein abhi bhi koi `todos` table nahi.

Blueprint ko actual table mein convert karne ke liye — **migration.**

---

## Migration Kya Hoti Hai

Prisma tumhara `schema.prisma` padhta hai, khud SQL likhta hai:

```sql
CREATE TABLE "todos" (
  "id" SERIAL PRIMARY KEY,
  "title" TEXT NOT NULL,
  "done" BOOLEAN NOT NULL DEFAULT false
)
```

Yeh SQL Neon pe run karta hai — table ban jaati hai.

Tumhe SQL nahi likhni. Prisma likhta hai.

---

## Migration Run Karo

```bash
npx prisma migrate dev --name init
```

- `migrate dev` — development mein migration run karo
- `--name init` — is migration ko naam do — pehli hai toh "init" theek hai

Kya hota hai andar:

1. `schema.prisma` padhta hai
2. SQL generate karta hai
3. Neon pe run karta hai — table ban jaati hai
4. `prisma/migrations/` mein history save hoti hai
5. **Automatically `prisma generate` bhi chalta hai** — ab `prisma.todo` available ho jaayega

Output:

```
✔ Generated Prisma Client
✔ Your database is now in sync with your schema.
```

## Agar `prisma.todo` Undefined Aaye Toh

Kabhi-kabhi migrate ke baad bhi `prisma.todo` undefined aata hai. Yeh Next.js cache ki wajah se hota hai.

### Problem
```
TypeError: Cannot read properties of undefined (reading 'findMany')
```

### Cause
Next.js purana Prisma client module cache kar leta hai — jisme Todo model nahi hota.

### Solution

1. **Server band karo** (Ctrl+C)

2. **Next.js cache clear karo:**
   ```bash
   # Windows PowerShell mein:
   Remove-Item -Recurse -Force .next
   
   # Ya fir:
   rm -Recurse -Force .next
   ```

3. **Prisma client dobara generate karo:**
   ```bash
   bunx prisma generate
   ```

4. **Server restart karo:**
   ```bash
   bun dev
   ```

Ab `prisma.todo` kaam karega kyunki naya generated client load hoga.
