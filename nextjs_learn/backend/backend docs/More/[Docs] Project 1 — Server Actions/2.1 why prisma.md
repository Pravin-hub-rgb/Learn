# Doc 2.1 — Why Prisma? Raw SQL Se Kya Dikkat Thi

---

## Abhi Tak Kya Hua

Doc 1.1 mein samjha — Server Actions se hum `TodoForm.tsx` mein hi ek function likh sakte hain jo database se baat kare. Koi alag file nahi, koi `fetch` nahi.

Par ek sawaal abhi bhi bacha hua hai —

> **"Database se baat karna matlab kya? Exactly kaise karte hain?"**

---

## Database Se Baat Karna Matlab?

Database ek alag program hai jo tumhare computer pe — ya cloud pe — chal raha hota hai. Usse baat karne ka ek hi tarika hai — **SQL likhna.**

SQL ek language hai jo database samajhta hai. Jaise:

```sql
-- Naya todo add karo
INSERT INTO todos (title, completed) VALUES ('Buy milk', false);

-- Saare todos lao
SELECT * FROM todos;

-- Todo complete karo
UPDATE todos SET completed = true WHERE id = 1;

-- Todo delete karo
DELETE FROM todos WHERE id = 1;
```

Yeh raw SQL hai. Directly database ko commands dena.

Toh sawaal yeh hai — **kya hum apne Next.js app mein yahi likhein?**

Technically haan, likh sakte hain. Par aao pehle dekhtey hain ki is approach mein kya kya dikkat aati hai.

---

## Raw SQL Ki Dikkat — Real Example Se Dekho

Maano tumhara `TodoForm.tsx` hai aur tum naya todo add karna chahte ho.

Raw SQL se kuch aisa dikhega:

```ts
// TodoForm.tsx

async function addTodo(title: string) {
  "use server"

  // Database se connection banana padega pehle
  const client = await db.connect()

  // Phir SQL likhna padega — string ke andar
  await client.query(
    'INSERT INTO todos (title, completed, created_at) VALUES ($1, $2, $3)',
    [title, false, new Date()]
  )

  // Connection band karna padega
  client.release()
}
```

Abhi yeh dekho —

**Pehli dikkat — SQL ek string hai.**

`'INSERT INTO todos (title, completed, created_at) VALUES ($1, $2, $3)'`

Yeh ek plain string hai. Agar tumne koi typo kiya — `todoos` likhा `todos` ki jagah — toh **code likhte waqt koi error nahi aayega.** Error tabhi aayegi jab app actually chalega aur koi todo add karne ki koshish karega. Tab ek cryptic error aayegi — "table todoos does not exist."

Matlab — **galti bahut baad mein pata chalti hai.**

**Doosri dikkat — columns yaad rakhne padte hain.**

Tumhe pata hona chahiye ki `todos` table mein exactly kaunse columns hain — `title`, `completed`, `created_at` — aur unka exact naam kya hai. Ek bhi column ka naam galat likha — wahi runtime error.

**Teesri dikkat — jo data wapas aata hai woh "any" type ka hota hai.**

```ts
const result = await client.query('SELECT * FROM todos')
const todos = result.rows // yeh kya hai? koi pata nahi TypeScript ko
```

`result.rows` mein jo data hai — TypeScript ko bilkul nahi pata ki usme kya hai. Koi `title` hai? Koi `completed` hai? TypeScript andhere mein hai. Toh tumhe manually batana padega — alag se types likhne padenge.

---

## Ek Aur Badi Dikkat — Har Database Ki SQL Alag Hoti Hai

Aaj tum PostgreSQL use kar rahe ho. Kal kisi reason se MySQL pe shift karna pada.

Raw SQL mein? **Bahut saari cheezein rewrite karni padti hain** — kyunki PostgreSQL aur MySQL ki SQL mein subtle differences hain.

Yeh ek alag problem hai — **tumhara code database pe dependent ho jaata hai.**

---

## Toh Chahiye Kya?

Soch ke dekho — raw SQL ki saari problems kya thi:

- String mein SQL likhna — typos pakde nahi jaate
- Columns yaad rakhne padte hain manually
- Wapas aane wala data untyped hota hai
- Database badla toh code badlega

Toh ek aisi cheez chahiye jo:

- SQL tumhare liye khud likhe — tumhe sirf batao "todo add karna hai"
- Columns woh khud jaane — tumhe yaad na rakhna pade
- Wapas aane wala data properly typed ho — TypeScript ko pata ho ki kya aa raha hai
- Database badlo — zyada kuch na badle

Yahi hai **ORM** — Object Relational Mapper.

---

## ORM Kya Hota Hai — Analogy Se Samjho

ORM ek **translator** hai — tumhare aur database ke beech.

Tum ORM ko bolte ho apni bhasha mein — "ek naya todo banao jiska title 'Buy milk' ho."

ORM khud SQL mein translate karta hai — `INSERT INTO todos (title) VALUES ('Buy milk')` — aur database ko bhejta hai.

Tumhe SQL yaad nahi rakhni. Tumhe columns yaad nahi rakhne. ORM sambhal leta hai.

---

## Prisma — Sabse Popular ORM Next.js Ke Liye

Bahut saare ORMs hain — Drizzle, TypeORM, Sequelize. Par **Prisma** Next.js ke saath sabse zyada use hota hai. Kyun?

Kyunki Prisma ne ek cheez bahut aachi ki — **tumhara database ka structure ek alag file mein likha jaata hai.** Ek baar. Clearly.

Woh file hoti hai — `schema.prisma`. Isme tum likhte ho — "mera todos table aisa dikhega":

```prisma
model Todo {
  id        Int      @id @default(autoincrement())
  title     String
  completed Boolean  @default(false)
  createdAt DateTime @default(now())
}
```

Bas yeh likhne ke baad — Prisma khud jaanta hai ki `Todo` mein kya kya hai. Aur tumhare poore app mein yeh information available ho jaati hai.

Ab `addTodo` function kaisa dikhega Prisma se?

```ts
// TodoForm.tsx

async function addTodo(title: string) {
  "use server"
  await prisma.todo.create({
    data: { title }   // bas itna — completed aur createdAt khud handle ho jaate hain
  })
}
```

Itna clean. Koi SQL string nahi. Koi columns manually nahi. Koi connection manage nahi.

## Side by Side — Fark Dekho

| Cheez | Raw SQL | Prisma |
|---|---|---|
| SQL likhni padti hai? | Haan — string mein | Nahi — Prisma likhta hai |
| Typo pakda jaata hai? | Nahi — runtime pe pata chalta hai | Haan — likhte waqt hi error |
| Data ka type pata hota hai? | Nahi — manually likhna padta hai | Haan — Prisma khud jaanta hai |
| Columns yaad rakhne padte hain? | Haan | Nahi — schema mein defined hai |
| Database badla toh? | Bahut kuch rewrite | Thoda sa config change |

---

## Summary

- Database se baat karne ka tarika hai SQL likhna — par raw SQL mein bahut dikkat hai
- Typos runtime pe pakde jaate hain, data untyped hota hai, columns yaad rakhne padte hain
- **ORM** ek translator hai — tum apni bhasha mein bolo, woh SQL mein translate karta hai
- **Prisma** sabse popular ORM hai Next.js ke liye — ek `schema.prisma` file mein apna database define karo, baaki Prisma sambhal leta hai
