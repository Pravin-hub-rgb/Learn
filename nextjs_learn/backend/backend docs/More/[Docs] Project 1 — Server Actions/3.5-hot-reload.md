# Doc 3.5 — Hot Reload Problem — Samjho Par Implement Mat Karo

---

## Abhi Tak Kya Hua

`lib/prisma.ts` ban gayi — simple, official pattern. Kaam kar raha hai.

Par development mein ek cheez hoti hai jo baad mein problem de sakti hai. Abhi implement nahi karenge — par samjhna zaroori hai taaki future mein samajh aaye.

---

## Problem Kya Hai

Next.js development mein **hot reload** hota hai — jab bhi koi file save karo, Next.js us module ko dobara load karta hai.

Abhi `lib/prisma.ts` mein yeh hai:

```ts
const adapter = new PrismaPg({ connectionString })
export const prisma = new PrismaClient({ adapter })
```

Yeh do lines **module level pe** hain — matlab jab bhi `lib/prisma.ts` reload hogi, yeh dobara chalenge. Naya `adapter`, naya `PrismaClient`, naya database connection.

Reload → naya connection.
File save → reload → naya connection.
Aur save → aur reload → aur naya connection.

Thodi der mein Neon ke paas **bahut zyada open connections** ho jaate hain. Neon free tier pe ek limit hoti hai — us limit ke baad Neon naya connection accept karna band kar deta hai — aur error aata hai.

---

## Fix Ka Idea Kya Hoga

Fix simple hai conceptually —

> "Pehli baar connection banao — ek global jagah save karo. Agli baar reload pe — pehle dekho 'kya pehle se connection hai?' — hai toh wahi use karo, naya mat banao."

Woh global jagah hoti hai **`globalThis`** — JavaScript mein ek special object jo poori app mein ek hi hota hai, reload pe reset nahi hota.

Aur TypeScript mein `globalThis` pe custom property add karne ke liye `as unknown as` casting use hoti hai — jo thodi complex hai.

Yeh sab **3.6 mein implement karenge** — tab tak ke liye 3.4 wala simple pattern theek hai. Development mein itni jaldi connections exhaust nahi honge beginner project mein.

---

## Ek Line Mein Summary

Hot reload → naya connection → zyada connections → Neon limit hit → error.
Fix → `globalThis` mein save karo → reload pe purana reuse karo.
